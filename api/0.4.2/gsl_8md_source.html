1 The Guideline Support Library (GSL)
2 ======================================
3 
4 Throughout the icedb source and headers, you might
5 have noticed that the code is peppered with objects
6 in the gsl namespace - entities like gsl::not_null and
7 gsl::span. The [Guideline Support Library](https://github.com/Microsoft/GSL)
8 is a library that contains functions and types that are
9 suggested for use by the [C++ Core Guidelines](https://github.com/isocpp/CppCoreGuidelines)
10 that are maintained by the [Standard C++ Foundation](https://isocpp.org/).
11 Icedb aims to follow these best practices, and adherence to these
12 guidelines should help ensure that the code is long-lasting and portable.
13 
14 The gsl headers are located in the icedb/compat/gsl subdirectory. They
15 are released under the MIT license, and are redistributed here because
16 they are not widely distributed to different platforms.

18 ## Summary of the Different GSL Templates and Macros in Icedb
19 
20 ### gsl::not_null<T>
21 
22 C and C++ pointers may be used:
23 - To pass a single object by address
24 - To indicate arrays
25 - To provide "output" parameters to functions
26 - To indicate "optional" parameters
27 
28 This is horribly vague, and it leads to programming mistakes.
29 gsl::not_null<T>(obj) ensures that obj is a non-null pointer.
30 It checks at runtime, and throws an error if a null pointer is encountered.
31 By placing this check in the declaration of a function, the check is
32 consistently performed. This is also an aid to debugging, whereby you
33 can easily ensure that a null pointer does not enter into a group of nested,
34 high performance functions.
35 
36 Of course, it does not check against passing an uninitialized pointer. Smart
37 pointers (or a smart compiler with all warnings enabled) can help mitigate this.
38 
39 ### gsl::span<T>
40 
41 This allows you to safely pass a range of objects to a function.
42 
43 - A common, unsafe, way to pass an array of objects to a function is to just use a
44 NULL-terminated pointer. But, what if you do not remember to ensure NULL-termination?
45 - You could also pass a pointer to the start of an object, along with a size parameter.
46 However, you the size variable is decoupled from the actual array, which allows for
47 mistakes.
48 
49 This is why C++ has iterators. However, iterators are usually employed only in standard
50 object containers. We could require that the passed objects be withing a standard container,
51 like a std::vector, but then we are forcing library users into our programming style.
52 Sometimes, a vector is not the appropriate container, and we want to avoid excessive
53 re-shuffling and copying of objects.
54 
55 A gsl::span<T> object has iterators, size(), begin() and end() functions. They are
56 trivially constructable.
57 
58 ### gsl::owner<T>
59 
60 Dynamically-allocated memory should be freed when it is no longer in use. Smart pointers,
61 like std::unique_ptr<T> and std::shared_ptr<T> provide facilities for freeing an object
62 when its last reference is gone. However, smart pointers are not always appropriate.
63 
64 gsl::owner<T> provides a way for a programmer to keep track of how ownership of a pointer
65 should transfer between function calls.
66 
67 ### Expects(...)
68 
69 This is a better version of assert. It is always useful to check assumptions, and using
70 Expects helps to clarify your intent to others who may read your code.
71 
72 ### Ensures(...)
73 
74 This is similar to Expects, but expresses an assert call as a post condition. That is, at the
75 end of a function, the condition must be respected or an error will occur.
